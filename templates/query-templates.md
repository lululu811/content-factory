# /query Skill 标准 Prompt 模板库

> **本文用途**：每个 MOC 配 3-5 个标准化 /query prompt，跑研究时直接套用，不用从零想。

**最后更新**：2026-06-20
**配合文档**：`templates/moc-routing.md`

---

## 一、通用 /query 模板

### 1.1 最新进展查询（5 分钟快查）

```
/query "在知识库中检索 {topic} 最近 30 天的 sources，列出 5-10 条最重要的，
按重要性排序，每条用 1 句话概括核心观点。
返回时标注每条的 source 文件名。"
```

**适用**：选题确定后，先快速了解库内有什么素材。

### 1.2 深度综合查询（30 分钟深挖）

```
/query "在知识库中检索 {topic} 的所有 sources、concepts、entities。
要求：
1. 列出 10-15 条核心 sources（按主题相关性排序）
2. 列出 5-10 个关键概念（用 [[wikilink]] 格式）
3. 列出涉及的 3-5 个实体（公司/人物/工具）
4. 总结这些 sources 的核心共识和分歧
5. 指出还有哪些方向 sources 覆盖不足"
```

**适用**：正式写作前，必须做的深度调研。

### 1.3 概念关系查询（建立知识图谱）

```
/query "{concept1} 和 {concept2} 在知识库中的关联是什么？
它们之间的产业链关系 / 因果关系 / 替代关系？
列出 3-5 条相关 sources 证明你的判断。"
```

**适用**：产业链分析时确认概念之间的关系。

### 1.4 时间线查询（事件演变）

```
/query "{topic} 在知识库中过去 6 个月的发展时间线是什么？
列出 5-8 个关键事件 / 政策 / 数据点，按时间排序。
每个事件标注日期和 source。"
```

**适用**：写复盘 / 趋势分析类文章。

## 四、查询结果使用流程

```
/query 跑出 sources 后：

1. 通读 5-10 条 sources 的核心观点
2. 提取每个 source 的核心数据点 + 证据等级
3. 检查 sources 之间是否有矛盾（矛盾处理：[[知识冲突]] 区块）
4. 写文章时引用 3-5 条最强 sources
5. 文章 frontmatter 的 tags 加上 MOC 名称
6. 写完后，把文章作为 source 同步到知识库（如果值得归档）
```

## 五、查询速度优化

| 模式 | 速度 | 深度 | 何时用 |
|---|---|---|---|
| 标题级 /query | 5 秒 | 浅 | 选题阶段快速判断 |
| 全文 /query | 30 秒 | 中 | 写作前核心查询 |
| 跨文件 /query | 2-5 分钟 | 深 | 重大选题深挖 |

**实操建议**：先用标题级判断库内有无相关素材，决定后再跑全文。
